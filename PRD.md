## 1. 文档目的

本文档作为《夜巡录：荒庙篇》的完整、可执行、可验收的游戏 PRD。它不是宣传文案，也不是单纯技术说明，而是给产品、策划、交互、美术、音频、测试、研发和后续商业化判断共同使用的统一需求基线。

本文档覆盖：

## 2. 一句话产品定义

《夜巡录：荒庙篇》是一款志怪题材的单人卡牌构筑 roguelike 第一关竖切 demo。玩家扮演“游方夜巡人”，在永宁县外的荒庙夜路中选择路线、经历战斗与怪事、获得卡牌和遗物、管理生命与金币，并最终击败 Boss 山君，让雾散天明。

## 3. 产品定位

### 3.1 当前定位

当前版本定位为 playable vertical slice，即“能完整跑通一关并验证核心体验的可玩竖切”。它已经不是静态概念稿，也不是纯 UI 原型，而是具备完整游戏闭环的第一章 demo。

当前已可验证的产品能力：

- 标题页选择难度。
- 程序化生成 8 层路线地图。
- 普通战、精英战、Boss 战可完成。
- 事件、休整、商店、删牌、升级可进入并产生局内影响。
- 战斗内支持抽牌、弃牌、洗牌、能量、格挡、符印、香火、虚弱、易伤、法门、消耗牌、状态牌。
- 胜利后进入结算过场，再进入奖励或胜利页。
- Web 静态部署，朋友可通过公开链接直接游玩。
- Electron 桌面打包能力保留。

### 3.2 竞品与类型参考

核心类型参考：

- 《Slay the Spire》：路线选择、公开敌人意图、卡牌构筑、遗物、事件和休整。
- 小体量 roguelike demo：短局、高密度反馈、以构筑方向而非剧情长度驱动重玩。

差异化方向：

- 题材不是西式奇幻或科幻，而是中国志怪、荒庙、城隍印、符箓、香火、阴市、山君。
- 核心机制从题材生长出来：符印是延迟与爆发，香火是临时法力，法门是战斗内常驻修法。
- 视觉方向偏暗场、纸质、朱砂、暗金、雾青，避免高饱和页游仙侠感。

## 4. 产品目标与非目标

### 4.1 当前版本目标

1. 让玩家在 10-25 分钟内完成一次第一关流程。
2. 让玩家理解“路线选择 → 战斗 → 奖励 → 构筑变强 → Boss 验证”的完整循环。
3. 让“符印、香火、抽牌循环、格挡生存、遗物联动”形成至少 2-3 种可感知构筑方向。
4. 让 demo 可以公开试玩，适合展示、录屏、朋友试玩和后续迭代反馈。
5. 让代码结构支持继续扩展卡牌、敌人、事件、遗物、章节和桌面客户端。

### 4.2 当前版本非目标

当前版本不以以下内容为验收目标：

- 多角色职业系统。
- 多章节长线剧情。
- 多敌人站位与群体战斗。
- 持久化存档、成就、账号系统。
- 每日挑战、排行榜、联机。
- 商业发行级素材授权闭环。
- 完整自动化平衡测试。
- Steam、App Store、国内小游戏平台正式发行。

## 5. 目标用户

### 5.1 核心用户

重度卡牌构筑 roguelike 玩家：

- 玩过《Slay the Spire》《Monster Train》《Griftlands》等游戏。
- 能理解费用、抽牌、弃牌、牌组污染、遗物联动、敌人意图。
- 关注构筑张力、数值合理性、路线收益与风险。
- 对“题材是否服务玩法”较敏感。

### 5.2 次级用户

志怪/国风题材兴趣用户：

- 被荒庙、阴市、城隍、符箓、香火、山君等题材吸引。
- 可能不熟悉卡牌构筑，但愿意通过演示难度体验。
- 更关注氛围、美术、文字、音效和完整感。

### 5.3 展示与传播用户

朋友试玩、作品集审阅者、潜在合作方：

- 希望点开网页即可开始。
- 希望 1-3 分钟内看到明确质感。
- 希望游戏不是“概念说明”，而是能真实操作。
- 对 bug、适配、卡顿、素材缺失容忍度低。

## 6. 核心体验支柱

### 6.1 夜路抉择

玩家每一层不是线性前进，而是在有限岔路中提前判断后续风险：现在打精英，还是找休整；进阴市花钱补强，还是走普通战拿奖励；事件是赌收益，还是绕开代价。

### 6.2 临时构筑

玩家不是带着完美套牌进局，而是在一局内逐步修正牌组。每次奖励选择都应带来明确问题：

- 当前缺伤害、缺防御、缺抽牌，还是缺爆发出口？
- 新牌是否会污染牌组？
- 是否应该跳过奖励保持牌组精简？
- 是否花金币删基础牌或状态牌？

### 6.3 公开压力

敌人意图公开。玩家每回合应清楚知道：

- 敌人这回合会攻击、格挡、强化、上 debuff，还是塞状态牌。
- 当前应防满、抢斩杀、继续叠符印，还是提前引爆。
- 选择吃伤害换收益是否值得。

### 6.4 符法爆发

构筑爽感围绕三条主线：

- 符印：叠在敌人身上的延迟伤害与爆发素材。
- 香火：战斗内临时资源，用于支撑五雷正法等爆发。
- 法门/遗物：改变抽牌、防御、开局资源、符印收益和出牌节奏。

### 6.5 克制志怪氛围

游戏应呈现“荒庙夜路”的阴冷感，而不是热闹仙侠。界面、音效、文字和素材都服务于“夜巡、雾、残香、纸灯、妖物、旧庙”的主题。

## 7. 用户旅程

### 7.1 标准局流程

1. 玩家进入标题页。
2. 选择难度：演示、标准或劫难。
3. 点击“开始夜巡”。
4. 进入加载页“雾门将开”。
5. 进入 8 层路线地图。
6. 从当前可选节点中选择路线。
7. 根据节点类型进入战斗、事件、休整或商店。
8. 战斗胜利后进入结算过场。
9. 普通/精英战进入奖励页，选择 1 张卡或跳过。
10. 回到地图继续前进。
11. 第 8 层进入 Boss 山君。
12. 击败山君后进入 Boss 结算过场。
13. 进入胜利页“雾散天明”。
14. 玩家可点击“再巡一夜”重新开始。

### 7.2 失败流程

1. 玩家在战斗、事件代价、状态牌阴寒等情况下生命降至 0。
2. 进入失败页“夜路尽头”。
3. 文案提示本局结束，但可重新开始。
4. 玩家点击“再巡一夜”开启新局。

## 8. 信息架构与屏幕需求

### 8.1 顶部 HUD

HUD 需要始终呈现当前局势核心信息：

- 主图标/印记。
- 玩家生命：`hp/maxHp`。
- 金币数量。
- 牌组张数。
- 遗物数量。
- 地图层数：`当前层/8`。
- 署名：`FF.INC × Codex 联合开发 · 仅供娱乐`。
- 回首页、静音、关于、重新开始按钮。

桌面端应保持横向紧凑布局。手机端应允许 HUD 分行和横向微滚动，避免挤压战斗区和产生横向页面溢出。

### 8.2 标题页

必须包含：

- 游戏名：荒庙夜巡。
- 题材描述。
- 三档难度选择。
- 开始按钮。
- 背景视频循环。

难度说明：

- 演示：血量更高、敌人更松、开局多一点循环支撑，适合录屏展示。
- 标准：完整路线体验，数值更稳，推荐默认选择。
- 劫难：敌人更硬更痛，用于挑战与压测。

### 8.3 路线地图页

必须包含：

- 8 层路线地图。
- 当前可选节点高亮。
- 已访问节点与路径反馈。
- 每个可选节点的卡片式说明。
- 夜巡札记日志。

交互要求：

- 只允许点击当前 `availableNodeIds` 中的节点。
- 不可选节点禁用。
- 节点 tooltip 或卡片必须说明收益/风险。
- 玩家应能提前看到至少一部分后续分岔，从而做路线规划。

### 8.4 战斗页

必须包含：

- Phaser 战斗舞台：背景、主角、敌人、雾气、受击晃动。
- React 战斗 UI：生命、格挡、香火、符印、意图、能量、手牌、牌库、弃牌堆、结束回合。
- 手牌扇形排布。
- 拖拽出牌目标提示。
- 出牌成功反馈。

交互要求：

- 攻击牌拖到敌人侧释放。
- 技能牌和法门牌拖到玩家侧释放。
- 能量不足、不可打出或状态牌应禁用。
- 结束回合按钮始终可见可点。
- 手机端不得出现核心按钮被手牌覆盖到不可用的情况。

### 8.5 结算过场页

普通战、精英战和 Boss 胜利后先进入 `cinematic` 屏幕：

- 展示敌人退散/伏诛标题。
- 播放对应结算视频。
- 如果视频缺失或加载失败，显示静态 fallback。
- 显示金币、遗物、待领取卡牌预览。
- 点击按钮或视频结束后进入奖励/胜利页。

### 8.6 奖励页

普通战和精英战后进入：

- 显示获得金币。
- 如果获得遗物，显示遗物名称和说明。
- 展示 3 张卡牌奖励。
- 可选择 1 张加入牌组。
- 可跳过卡牌，保持牌组精简。
- 显示夜巡札记日志。

### 8.7 怪事页

事件页需要：

- 事件标题和叙事文本。
- 两个选择按钮。
- 每个选择明确标注收益与代价。
- 选择后立即执行效果并回到地图或进入删牌/升级选择。

### 8.8 休整页

休整页提供两个选择：

- 静坐调息：回复最大生命 30%。
- 朱砂重描：升级 1 张未升级且非状态牌。

选择后返回地图。

### 8.9 商店页

商店页包含：

- 3 张随机卡牌商品。
- 1 个随机未拥有遗物商品。
- 1 个删牌服务。
- 离开阴市按钮。

价格规则：

- 普通卡：48 金。
- 非普通卡/稀有度 common：48 金。
- uncommon 卡：68 金。
- rare 卡：95 金。
- 遗物：150 金。
- 商店删牌：75 金。

金币不足或已售商品必须禁用。

### 8.10 关于页

关于页必须说明：

- 项目由 FF.INC 与 Codex 联合开发。
- 当前为娱乐、学习和非商业展示。
- 素材授权与商业化边界需要后续确认。
- 可返回原页面或回到首页。

## 9. 核心玩法系统

### 9.1 宏观路线系统

当前路线共 8 层：

| 层级 | 地名 | 当前节点构成 |
| --- | --- | --- |
| 1 | 县口 | 2 个普通战 |
| 2 | 荒村 | 普通战、事件、普通战，顺序洗牌后按 lane 排布 |
| 3 | 井边 | 精英、事件、商店，顺序洗牌后按 lane 排布 |
| 4 | 破庙 | 休整、普通战、事件，顺序洗牌后按 lane 排布 |
| 5 | 林道 | 精英、普通战、事件，顺序洗牌后按 lane 排布 |
| 6 | 阴市 | 商店、休整、普通战，顺序洗牌后按 lane 排布 |
| 7 | 山门 | 精英、休整、事件，顺序洗牌后按 lane 排布 |
| 8 | 正殿 | Boss |

连线规则：

- 每个节点连接下一层中 lane 距离不超过 2 的节点。
- 如果没有相邻节点，则连接最近的 2 个 fallback 节点。
- 玩家只能选择当前可达节点。

设计意图：

- 第一层只给普通战，让玩家先进入战斗学习。
- 第 3、5、7 层提供精英机会，形成风险收益路线。
- 商店和休整穿插在中后段，让金币和生命管理产生决策。
- Boss 固定在第 8 层，形成明确终点。

### 9.2 战斗循环

战斗采用单敌人、回合制、公开意图模式。

开战流程：

1. 根据节点类型选择敌人。
2. 按难度与楼层修正敌人生命。
3. 复制玩家牌组为抽牌堆并洗牌。
4. 清空玩家战斗内状态：格挡、香火、虚弱、易伤、法门。
5. 触发开战遗物。
6. 选择敌人意图。
7. 开始玩家第 1 回合。

玩家回合开始：

1. 回合数 +1。
2. 本回合出牌计数清零。
3. `attackPlayed` 清零。
4. 玩家格挡清零。
5. 能量重置为 `maxEnergy`。
6. 第 1 回合触发小纸马、城隍小印、旧雨伞、夜明砂等遗物。
7. 触发法门“请城隍”。
8. 抽牌：基础 5 张 + 夜眼加成 + 夜明砂首回合加成。

玩家出牌：

1. 检查手牌中是否存在对应 uid。
2. 检查是否不可打出。
3. 检查费用是否为数字且玩家能量足够。
4. 扣除能量。
5. 从手牌移除。
6. 结算卡牌效果。
7. 触发遗物：桃木剑穗、破香炉、无字残页等。
8. 根据牌类型进入弃牌堆或消耗堆。
9. 如果敌人生命 <= 0，进入胜利流程。

结束回合：

1. 检查手牌中的阴寒数量，每张导致失去 2 点生命。
2. 未死亡则将手牌移入弃牌堆或消耗堆。
3. 进入敌人回合。

敌人回合：

1. 先触发符印灼烧。
2. 若敌人死亡，进入胜利流程。
3. 执行公开意图。
4. 衰减敌人虚弱、易伤。
5. 衰减玩家虚弱、易伤。
6. 若玩家死亡，进入失败页。
7. 选择下一次敌人意图。
8. 开始下一玩家回合。

### 9.3 资源系统

| 资源 | 作用 | 生命周期 | 当前数值策略 |
| --- | --- | --- | --- |
| 生命 | 玩家局内生存资源 | 跨战斗保留 | 通过难度、事件、休整调节压力 |
| 能量 | 每回合打牌费用 | 每回合刷新 | 标准/劫难 3，演示 4 |
| 格挡 | 抵消攻击伤害 | 每回合清空 | 鼓励当回合攻防决策 |
| 香火 | 战斗内临时法力 | 战斗内保留，开战清零 | 支撑五雷正法和请城隍链条 |
| 金币 | 买牌、买遗物、删牌 | 跨战斗保留 | 战斗奖励、事件、狐市旧钱提供 |
| 符印 | 敌人身上的标记 | 战斗内保留，每次敌回合递减 1 | 延迟伤害，也可被焚符爆发 |

### 9.4 状态系统

| 状态 | 作用 | 当前实现 |
| --- | --- | --- |
| 符印 | 敌人回合开始造成 `层数 * 3` 伤害，雷击木后每层 +1，然后层数 -1 | 可由朱砂符、太上敕令、照妖镜、青铜古镜施加 |
| 虚弱 | 被虚弱单位造成伤害时按 75% 结算 | 玩家弱化敌人，敌人也可令玩家虚弱 |
| 易伤 | 受攻击伤害提高到 150% | 当前由照妖镜施加给敌人 |
| 力量 | 敌人攻击每 hit 增加固定伤害 | 山魈、术士、山君可强化 |
| 法门 | 本场战斗持续能力 | 夜眼、请城隍 |
| 阴寒 | 无法打出；回合结束若在手牌，失去 2 点生命 | 敌人 curse、事件、翻经可加入 |

## 10. 难度策略

当前共有三档难度：

| 难度 | 玩家生命 | 初始金币 | 每回合能量 | 敌人生命倍率 | 敌人伤害倍率 | 金币奖励倍率 | 起始遗物 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | --- |
| 演示 story | 96 | 90 | 4 | 0.82 | 0.72 | 1.25 | 旧雨伞、无字残页 |
| 标准 normal | 84 | 65 | 3 | 0.92 | 0.90 | 1.10 | 无 |
| 劫难 hard | 74 | 45 | 3 | 1.08 | 1.08 | 1.00 | 无 |

演示难度额外规则：

- 初始牌组中的朱砂符升级。
- 初始牌组中的回风符升级。

设计判断：

- 演示难度用于录屏、传播和新手试玩，允许玩家更容易看到构筑链条。
- 标准难度用于默认体验，敌人并非满血满伤害，保证第一章 demo 不因随机性过早劝退。
- 劫难难度用于重度玩家和策划压测，资源更紧、敌人压力更高。

## 17. 数值策略

### 17.1 基础伤害与防御基准

当前标准难度下，初始能量为 3。基础牌参考：

- 斩妖剑：1 费 6 伤害，升级后 9。
- 护身符：1 费 5 格挡，升级后 8。
- 朱砂符：1 费 4 伤害 + 1 符印，升级后 6 伤害 + 2 符印。

因此标准初始牌组单回合理论基础输出约为 12-18，基础防御约为 5-15，视抽牌而定。普通怪标准难度第一层 HP 约 39-44，意味着玩家通常需要 3-5 回合解决第一战。

### 17.2 符印策略

符印是“持续伤害 + 爆发素材”的双用途资源。

基础结算：

```text
敌人回合开始：伤害 = 符印层数 * 3
若有雷击木：伤害 = 符印层数 * 4
结算后符印层数 -1
```

爆发结算：

```text
焚符基础：移除全部符印，每层 5 伤害
焚符升级：每层 7 伤害
```

策略含义：

- 低层符印适合配合回风符、引雷、破邪、照妖镜等联动。
- 高层符印适合焚符一次爆发。
- 雷击木提高持续结算收益，而不是直接提高焚符收益。
- 青铜古镜让每场战斗开局自带符印条件，显著提升符印牌稳定性。

### 17.3 香火策略

香火用于支撑法术爆发，主要来源：

- 金光咒：+1。
- 香火续灯：+2/+3，消耗。
- 请城隍：每回合 +1。
- 城隍小印：第 1 回合 +2。

主要出口：

- 五雷正法：消耗全部香火，造成 `12/16 + 香火 * 5/6` 伤害。

设计含义：

- 香火不是每回合刷新资源，而是战斗内积累资源。
- 玩家需要判断是否提前五雷收割，还是继续积攒香火。
- 请城隍和城隍小印可以把香火构筑从偶发爆发变成稳定主轴。

### 17.4 抽牌与循环策略

核心循环来源：

- 0 费抽牌：清心咒、回风符。
- 抽防一体：云步、驱雾。
- 弃牌堆回卷：灰烬回卷。
- 常驻抽牌：夜眼。
- 每 3 张牌抽牌：无字残页。

策略含义：

- 当前奖励池强制保底循环牌，是为了让玩家更容易看到构筑滚动。
- 0 费牌与无字残页形成出牌链。
- 灰烬回卷降低小牌组抽空后的断档风险。
- 翻经提供强抽牌但加入阴寒，属于高收益高污染。

### 17.5 牌组污染策略

阴寒是当前唯一状态牌：

- 费用为 `-`，无法打出。
- 回合结束时若仍在手牌，每张失去 2 生命。

来源：

- 灯笼精、庙祝尸、黑坛术士的 curse 行为。
- 翻经。
- 破庙神像取钱。
- 无面书生复制。

设计含义：

- 阴寒不是立即伤害，而是抽牌与手牌管理压力。
- 抽牌越强，越可能把阴寒抽上手。
- 删牌服务和跳过奖励间接提升抗污染能力。

### 17.6 敌人压力曲线

当前压力由三部分叠加：

1. 层数 HP scaling：越往后敌人生命越高。
2. 难度倍率：统一影响 HP 与伤害。
3. 敌人行为：普通怪教学，精英/Boss 综合考验。

设计建议：

- 标准难度下，普通战应允许大部分初始构筑 3-6 回合获胜。
- 精英应迫使玩家提前准备防御或爆发，否则付出明显生命代价。
- Boss 应考验成型度：没有抽牌、格挡或爆发链条时难以稳定通过。

## 18. 交互设计要求

### 18.1 拖拽出牌

交互目标：

- 让出牌具备实体卡牌的“推出去”手感。
- 让攻击和自我加护目标明确。
- 避免点击误操作。

规则：

- 攻击牌只接受敌人侧释放。
- 技能/法门只接受玩家侧释放。
- 中央过窄区域不判定目标，减少误放。
- 拖拽时显示目标幽灵和释放提示。
- 成功释放后显示 burst 反馈。

### 18.2 地图选择

地图需要支持两种阅读方式：

- 左侧全局路径图：看未来路线。
- 可选节点卡片：读当前选择收益。

当前节点应通过颜色、亮度和可点击状态明确区分。

### 18.3 响应式与移动端

当前公开网页需要支持手机访问。移动端优先目标不是“最佳竞技操作”，而是“能完整清晰展示并能开始游戏”。

移动端要求：

- 不产生页面横向滚动。
- 页面切换时回到顶部。
- HUD 不遮挡核心操作。
- 地图可纵向滚动。
- 战斗页核心信息、结束回合、能量、手牌至少在主流 375px 宽屏幕可见。
- Phaser 舞台在窄屏隐藏重复文字，避免与 React 状态面板叠字。

桌面端要求：

- 1920×1080 下保持完整清晰展示。
- 战斗画布铺满主区域。
- 手牌、结束回合、牌堆不互相遮挡。

## 19. 视听需求

### 19.1 美术方向

视觉关键词：

- 志怪、荒庙、夜雾、残香、纸灯、城隍、朱砂、旧铜、雾青。

当前资源类型：

- `assets/generated/characters/`：主角立绘。
- `assets/generated/enemies/`：敌人位图。
- `assets/generated/backgrounds/`：战斗背景与循环视频。
- `assets/generated/cinematics/`：胜利结算视频与 poster。
- `assets/vendor/shushan/`：UI 图标、卡框、费用宝石等。
- `assets/audio/`：背景音乐与音效。

视觉要求：

- 主角和敌人轮廓必须清晰。
- UI 不应压过战斗舞台主体。
- 卡牌费用、类型、名称、效果必须清楚可读。
- 卡面图标应辅助理解类型，而非占满文字空间。

### 19.2 音频方向

当前音频模式：

- 标题页 BGM 音量更高。
- 游戏内 BGM 音量降低，作为氛围底色。
- 点击、卡牌、攻击、火焰、雷击、蓄力、奖励、危险提示通过 `lastFx` 触发。

音效目标：

- 点击有反馈。
- 打牌有纸页/气功感。
- 攻击与受击有重量。
- 火符和雷法有区别。
- 奖励音效形成正反馈。

## 20. 技术需求

### 20.1 技术栈

- React 18：UI、页面流转、拖拽交互。
- Phaser 3：战斗舞台、角色敌人、雾气粒子、震动反馈。
- TypeScript：状态、内容和规则类型。
- Vite：开发与静态构建。
- Electron：桌面客户端包装。
- Cloudflare Pages：公开静态部署。

### 20.2 架构原则

当前架构分层：

- `src/game/content.ts`：内容数据，包含卡牌、敌人、遗物、事件、节点文案。
- `src/game/engine.ts`：游戏规则状态机。
- `src/game/types.ts`：类型定义。
- `src/App.tsx`：React UI、屏幕切换、交互入口。
- `src/phaser/CombatStage.tsx`：战斗舞台视觉投影。
- `src/styles.css`：视觉样式与响应式。

核心原则：

- React 是唯一游戏状态来源。
- Phaser 不持有完整规则状态，只接收 snapshot 并做视觉投影。
- Engine 不直接操作 DOM 或 Audio，只通过 `lastFx` 输出音效信号。
- 当前卡牌效果用手写分支实现，适合 demo 快速迭代；未来内容扩张后可升级为 DSL 或表格驱动。

### 20.3 部署需求

Web 部署：

- 项目必须能通过 `npm run build` 生成静态 `dist/`。
- `dist/` 中图片、视频、音频必须完整打包。
- Cloudflare Pages 可直接托管。
- 正式链接为 `https://card-game-night-patrol.pages.dev/`。

桌面部署：

- Electron 入口为 `electron/main.cjs`。
- 支持 `npm run desktop` 本地预览。
- 支持 `npm run desktop:pack` 和 `npm run desktop:dist` 打包。

## 21. 验收标准

### 21.1 功能验收

P0 必须通过：

- 标题页可打开。
- 三档难度均可开始游戏。
- 地图可选择节点。
- 普通战可完成。
- 精英战可完成。
- 事件选项可结算。
- 休整回血/升级可结算。
- 商店买牌/买遗物/删牌/离开可结算。
- 奖励页可选牌或跳过。
- Boss 可进入并可胜利。
- 失败页可出现并可重新开始。
- 胜利页可出现并可重新开始。

### 21.2 数值验收

建议手测场景：

- 标准难度不开挂，从第一层走到 Boss，至少 60% 以上合理路线可通关。
- 演示难度应显著更容易，适合录屏展示。
- 劫难难度应更紧张，但不应第一场普通战频繁无解。
- 符印构筑应能通过焚符/引雷形成可感知爆发。
- 香火构筑应能通过五雷正法形成高伤害终结。
- 抽牌循环构筑应能明显提高一回合出牌数量。
- 精英奖励的遗物必须值得承担风险。
- 跳过卡牌在部分情况下应是合理选择。

### 21.3 UI/UX 验收

桌面：

- 1920×1080 下 HUD、地图、战斗、手牌完整展示。
- 无关键按钮被遮挡。
- 卡牌文字可读。
- 过场视频或 fallback 可见。

手机：

- 375×667 和 390×844 下可进入游戏。
- 地图页可读可滚动。
- 战斗页无横向页面溢出。
- 结束回合按钮可见可点。
- 生命、能量、敌人意图和手牌不互相严重遮挡。

### 21.4 构建验收

每次发布前至少执行：

```bash
npm run build
```

通过标准：

- TypeScript 编译通过。
- Vite 构建成功。
- `dist/` 产物存在。
- 无缺失资源导致运行时空白。

## 22. 测试策略

### 22.1 当前测试方式

当前仓库没有自动化测试框架。主要验证方式为：

- `npm run build` 编译构建。
- 浏览器手测核心路径。
- 移动端和桌面端视口人工/浏览器自动化检查。
- Cloudflare 线上链接复验。

### 22.2 推荐测试用例

基础路径：

1. 标准难度开始 → 选择普通战 → 打出攻击/技能/法门 → 胜利 → 领取奖励 → 回地图。
2. 进入事件 → 每个事件选项至少触发一次。
3. 进入休整 → 回血和升级各触发一次。
4. 进入商店 → 买卡、买遗物、删牌、金币不足状态各验证一次。
5. 进入精英 → 胜利后确认必得遗物。
6. 进入 Boss → 胜利后确认进入胜利页。
7. 通过生命损失触发 gameover。

数值路径：

- 验证朱砂符、太上敕令、照妖镜可施加符印。
- 验证符印在敌人回合开始前造成伤害并递减。
- 验证焚符移除所有符印并造成层数伤害。
- 验证雷击木提高符印每层持续伤害。
- 验证香火续灯、金光咒、请城隍、城隍小印提供香火。
- 验证五雷正法消耗全部香火。
- 验证阴寒在手牌结束回合时扣血。
- 验证夜眼每回合多抽 1。
- 验证无字残页每 3 张牌抽 1。

适配路径：

- 1920×1080 桌面。
- 1366×768 笔记本。
- 390×844 手机。
- 375×667 小屏手机。

## 23. 当前风险与技术债

### 23.1 产品风险

- 第一章内容量仍偏小，重玩深度依赖当前卡池和遗物数量。
- 战斗只有单敌人，长期扩展后可能缺少站位和目标选择深度。
- 事件数量少，重复游玩会较快见底。
- Boss 前缺少更强剧情铺垫和入场仪式。

### 23.2 数值风险

- 当前没有自动化平衡模拟，难度体感依赖手测。
- 随机敌人意图完全随机，可能出现连续高压或连续低压。
- 奖励池保底提高了可玩性，但会降低纯随机 roguelike 的不可控感。
- 香火和符印两条爆发线后续扩卡时容易失控。
- 阴寒扣血较低，若后续抽牌更强，可能需要重新评估污染强度。

### 23.3 技术风险

- 卡牌效果手写分支，扩展到大量卡牌后维护成本上升。
- 没有单元测试，规则回归风险较高。
- 视频资源体积较大，加载和国内访问速度需要继续观察。
- 素材授权边界需要商业化前逐项确认。
- 移动端可玩但不是最佳体验，拖拽手感仍需更多真实设备验证。

## 27. 总结

《夜巡录：荒庙篇》当前已经具备一个小型卡牌构筑 roguelike 第一关 demo 的核心骨架：路线、战斗、奖励、事件、商店、休整、Boss、胜败、视听和公开部署。它的核心优势是题材与机制已经有初步互相支撑的关系：符印、香火、法门、阴寒、城隍印都不是纯氛围词，而是能进入数值系统和操作决策的玩法组件。

下一阶段的关键不是推翻重做，而是在当前骨架上继续补密度：更多内容、更稳数值、更好触控、更强事件表现和更可靠测试。只要继续保持“题材服务玩法、玩法反哺题材”的方向，这个 demo 有机会从可玩原型升级为有辨识度的独立卡牌游戏雏形。
